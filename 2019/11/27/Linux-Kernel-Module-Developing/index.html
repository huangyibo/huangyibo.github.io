Linux 内核模块开发实践与内核编译

每周技术分享 Linux Kernel LKM Kernel Module 2019/11/27

摘 要

鉴于最近需要改进Linux内核协议栈的BBR协议，便针对Linux内核模块的运行开发过程、内核编译进行了探索性实践，记录一下具体的实践过程，希望对于想要入门Linux内核开发却因庞大的代码树望而却步的童鞋们一些入门实践性的建议。本文主要介绍了开发Linux内核模块的主要流程、Linux内核编译与安装、用户空间程序与内核模块之间基于device file的通信实践三个部分。

0x01. A sample for Linux Loadable Kernel Module (LKM)

内核模块(Kernel Module)是可以按需加载到内核或从内核卸载的代码实现，在不需要重启系统的情况下扩展基础内核的功能。自定义内核模块可以通过两种方式加载到内核中：

  • 直接基于Linux内核源码树增加新的代码模块，并重新编译Linux内核；
  • 在Linux系统运行时增加新的内核模块，这种方式更方便有效。

在第二种方式中，由于在Linux系统运行时加载新的内核模块代码(这些代码实现不是官方Linux Kernel所含有的)，这些新内核模块被称为”Loadable Kernel Module（LKM）”，不同于基内核(“Base Kernel”)。基内核Base Kernel位于/boot目录下，在系统启动引导过程中被加载；而LKM是在Base Kernel已经加载运行之后被加载的。尽管如此，LKM仍然是Linux内核的一部分，并与Base Kernel通信以完成其功能。

LKMs可以实现很多功能，主要归为三类：

  • 设备驱动(Device Driver)
  • 文件系统驱动(Filesystem driver)
  • 系统调用(System calls)

LKMs不是用户空间程序，而是内核的一部分。以下内容通过一个简单内核模块的实现来介绍如何开发一个内核模块。

0x03. 开发相对复杂的Linux内核模块

在第一节中我们介绍了一个简单的Hello World的内核模块的开发过程，下面会以一个相对复杂的案例介绍如何开发一个相对复杂的Linux内核模块。

内核模块可以完成各种各样的任务，其中与用户空间的应用交互通信是最常见的场景之一。而由于用户空间应用没有直接访问内核空间内存的权限，它们需要通过使用API与内核空间模块交互。在技术实现上有多种可行的方法，最通用的方式是创建一个设备文件(Device file)。在Linux系统中，使用/dev/null、/dev/zero等类似的命令是与名为zero、null的设备(device)在交互通信，以获得期望得到的数据或响应。

在我们的案例中，与我们实现的内核模块设备(Device)交互(主要是read操作)会返回"Hello World"，同时对于设备文件的write、open和release操作实现简单的handler function响应。我会尽可能的展示出通过device file响应用户空间应用的过程。
